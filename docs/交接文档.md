## 补充更新（2025‑11‑08）

本次补充迭代聚焦“默认 OBJ 流程、直链推理修复、OBJ 在线预览与加载体验”。要点如下：

- 供应商直链推理（同任务多格式）的健壮性修复
  - 场景：供应商默认返回 `0.obj.zip` 或 `*.obj` 时，需要“推理”出同目录下其它格式直链（glb/stl/fbx）。
  - 修复：统一剥离双扩展（如 `.obj.zip`），仅在请求 OBJ 时才尝试 `base.zip`；其它格式优先 `base.<fmt>.zip` → `base.<fmt>`；避免把 `base.zip`（OBJ 包）误当作 GLB。
  - 影响文件：
    - `app/api/assets/[uuid]/renditions/route.ts:buildCandidateUrls`
    - `app/api/hitem3d/callback/route.ts:buildCandidateUrls`
    - `app/api/cron/hitem3d-auto-finalize/route.ts:buildCandidateUrls`

- OBJ 文件清单接口增强（支持从 R2 内置 zip 解包）
  - `GET /api/assets/:uuid/renditions/files?format=obj`：若仅存在 `file.obj.zip` 或仅有 `.obj`，会：
    1) 优先拉取供应商 zip；
    2) 否则从 R2 中已落库的 `file.obj.zip` 解包到 `assets/<user>/<asset>/obj/`；
    3) 只返回可渲染文件（`.obj/.mtl/.png/.jpg/.jpeg/.webp`）。
  - 全部清单条目按 `Content‑Disposition=inline` 签名，确保浏览器子资源可直接加载（不触发下载）。
  - 调试：可在请求参数中加 `debug=1` 获取 `steps/uploads`（已脱敏）。
  - 影响文件：`app/api/assets/[uuid]/renditions/files/route.ts`（新增 zip 解析、R2 解包与 inline 签名）。

- 仅保留 OBJ 派生（前端层面）
  - `components/assets/RenditionsPanel.tsx` 现阶段仅暴露 OBJ，隐藏 GLB/STL/FBX 的“生成与下载”按钮（后端仍兼容）。

- 资产详情页改为 OBJ 在线预览
  - 新增：`components/assets/ViewerOBJ.tsx`（Three.js + OBJLoader + MTLLoader）
    - 通过 npm 方式打包 `three` 与 `examples/jsm/*`，彻底移除对 unpkg 的依赖；避免 CDN/CSP 造成的初始化失败。
    - MTL 设置 `resourcePath`，URL 映射大小写/基名容错，纹理/MTL/OBJ 均走签名直链。
    - sRGB 输出、改良灯光；相机自动取景（完整框选模型）。
    - 兜底贴图：若 MTL 未命中任何 `map_*`，会将清单里的第一张图片应用到 Mesh（保证不是灰模，便于用户确认成品）。
    - 加载进度覆盖层（仅覆盖预览区域）：
      - 采用“目标进度 + 缓动逼近”的丝滑动画；
      - LoadingManager 阶段：10→90→95；`onLoad` 解析：98；首帧渲染：100；
      - 进度到 100% 后 150ms 再揭罩显示模型，避免闪烁。
    - Debug 模式：默认隐藏调试文案，访问详情页时在 URL 加 `?debug=1` 或 `?objdebug=1` 可显示“加载开始/加载中/解析中/就绪”等详细状态。
  - 新增：`components/assets/AssetAutoPreviewOBJ.tsx`（调用 files 接口物化并下发清单，透传 debug 开关）。
  - 页面：`app/[locale]/(default)/(console)/my-assets/[uuid]/page.tsx` 替换为“OBJ 预览”。

- 构建与依赖
  - 依赖：`three@^0.160.x`、`@types/three@^0.160.x`。
  - Vercel：新增 `shipany-template/vercel.json`，安装命令改为 `pnpm install --no-frozen-lockfile`，以便 CI 安装新增依赖与类型定义。

## 补充更新（2025‑11‑09）供应商 ZIP 落库与 OBJ 预览修复（本次对话）

背景：实测“人像任务”常返回带鉴权参数的 ZIP 直链（如 `0.zip?auth_key=...&t=...`）。浏览器点击可直接下载，但服务端落库偶发 `403`，导致资产未入库、详情页 OBJ 预览不可用。

根因与思路：
- 并非“人像 vs 非人像”的业务差异，而是不同产物分发域名/策略差异。此前我们默认发送 `Referer/Origin` 可能触发部分 CDN 的防盗链；而浏览器直链下载通常不带固定 Referer，反而能通过。
- 直链可能回 `*.zip` 或 `*.obj.zip`。原实现仅识别 `file.obj.zip`，未覆盖 `file.zip` 的解包路径。

修复要点：
- 放宽供应商下载请求头（避免无谓的 403）：
  - 不再强制发送 `Referer/Origin`（除非显式配置 `HITEM3D_REFERER`）。
  - 统一添加浏览器常见请求头：`User-Agent=Mozilla/5.0`、`Accept=*/*`、`Accept-Language=zh-CN,zh;q=0.9,en;q=0.8`。
- finalize/回调/兜底导出全链路增强：
  - 忽略“封面下载失败”，不再中断“主文件入库”。
  - 当 queryTask 返回的 `url` 不可访问时，自动回退尝试同目录 `*.obj.zip` 与 `*.zip`（命中则保存为 `file.zip`）。
- OBJ 文件清单端点增强：
  - 除了供应商 ZIP 直通外，可从 R2 中已存在的 `file.zip` 或 `file.obj.zip` 自动解包到 `assets/<user>/<asset>/obj/`，并对 `.obj/.mtl/贴图` 统一 `inline` 签名，确保子资源可被 Three.js 直接加载。
- UI 提示修正：
  - 列表页仅对 GLB 提供临时直链预览；非 GLB 场景提示引导“去详情页做 OBJ 在线预览”；并修复“带 query 的直链扩展名识别”。
- 诊断工具新增：
  - `GET /api/debug/vendor-fetch?url=...`：服务器直连供应商 URL 的探测（返回 status/ok/headers/size），便于快速定位 `403/网络差异`。
  - `GET /api/debug/hitem3d/finalize?task_id=...`：一键触发 finalize 兜底（浏览器即可操作）。

影响文件：
- 头部策略与容错：
  - `app/api/hitem3d/callback/route.ts`
  - `app/api/hitem3d/finalize/route.ts`
  - `app/api/assets/[uuid]/renditions/route.ts`
  - `app/api/cron/hitem3d-auto-finalize/route.ts`
- OBJ 清单/解包：
  - `app/api/assets/[uuid]/renditions/files/route.ts`
- 调试端点：
  - `app/api/debug/vendor-fetch/route.ts`
  - `app/api/debug/hitem3d/finalize/route.ts`
- UI：
  - `components/assets/TaskStatus.tsx`

快速验证：
1) 服务器是否能拉到供应商 ZIP：
   - 浏览器登录后访问 `GET /api/debug/vendor-fetch?url=<encodeURIComponent(供应商直链)>`，期望 `status=200`。
2) 一键 finalize（入库）：
   - `GET /api/debug/hitem3d/finalize?task_id=...` 返回 `{ code:0, data:{ asset_uuid } }`。
3) 详情页 OBJ 预览：
   - `/my-assets/<asset_uuid>?objdebug=1` 看到加载进度→渲染模型。
4) 若仍未就绪：
   - `GET /api/assets/<asset_uuid>/renditions/files?format=obj&debug=1` 查看物化/解包步骤与签名文件清单。

可选配置：
- 本次修复不依赖 `HITEM3D_APPID/HITEM3D_REFERER`；如供应商提供固定 Referer 要求，可补上 `HITEM3D_REFERER` 以提升一致性。

已知风险与边界：
- 供应商直链有效期有限（带鉴权参数），应尽快 finalize 入库。
- 个别地域/网络差异仍可能造成服务端偶发失败；此时可借助“调试端点 + 再次尝试”，或后续引入“浏览器辅助入库（前端直传 R2）”。

备注：本次修复已在预发域验证“人像 ZIP 直链 → 入库 → OBJ 预览”闭环可用。

### 2) 帐密登录误判失败（但实际已登录）的修复
现象：邮箱登录接口返回 401（前端弹窗提示“帐密错误”），但随后访问“资产/创作”等页面显示为已登录。

原因：在部分环境下，Auth.js/NextAuth 会在 Route Handler 内部通过 `next/headers` 的 `cookies()` 直接写入会话 Cookie；此时 `signIn()` 不一定返回 `Response` 或 `ok=true`，导致我们只看返回值会误判失败。

修复（/api/auth/login 与 /api/auth/register）：
- 视以下任一条件为成功：
  - `signIn()` 返回 `Response` 且 `ok` 或 3xx；
  - `Response` 返回头中包含 `Set-Cookie`，并转发给客户端；
  - “已处于登录态”（登录前检查 `auth()`）；
  - 登录前后对比 `cookies()` 快照，若发现会话 Cookie 名称变更/新增（`authjs.session-token`、`__Secure-authjs.session-token`、`next-auth.session-token`、`__Secure-next-auth.session-token`），则判定为成功。
- 兼容 Next 15：`cookies()` 可能为 Promise，增加 `getCookieMap()` 辅助函数进行异步快照。

主要改动文件：
- `app/api/auth/login/route.ts`：成功判定标准统一；新增“登录前后 Cookie 快照对比”；已登录兜底为成功；登录后再次兜底检查 `auth()`。
- `app/api/auth/register/route.ts`：注册后的自动登录同样透传 `Set-Cookie`，并加入 Cookie 变更检测。

### 三、封面策略 v1：供应商封面 & 用户上传输入图（后台自动处理）

> 本期实现：**首次封面 = 供应商封面优先，其次用户上传输入图**；  
> 后续目标：**封面优先级 = 预览截图 > 供应商封面 > 用户上传输入图**（截图封面将在后续迭代中补充）。

- 在提交阶段保存输入封面候选：`app/api/hitem3d/submit/route.ts`
  - 从表单中解析：
    - 单图模式：`images[0]`；
    - 多视图模式：当前版本取 `multi_images[0]` 作为“前视图”候选（若后续有更清晰前视图字段，可调整）。
  - 在 `submitTask` 成功返回 `task_id` 后，异步保存该候选图：
    - 使用 `newStorage().uploadFile`：
      - key：`assets/<user_uuid>/input-covers/<task_id>.<ext>`（`ext` 根据文件名/MIME 推断：png/webp/jpg）。
      - `ContentDisposition='inline'`，`ContentType` 按格式匹配（`image/png` / `image/webp` / `image/jpeg`）。
    - 失败时仅记录日志，不影响任务提交流程。
  - 测试：`tests/integration/hitem3d-submit-route.test.ts`：
    - 成功路径用例中增加断言，验证 `uploadFile` 被调用一次且 key 形如 `assets/u-1/input-covers/task-123.*`。

- 在 callback 成功时确定首次封面：`app/api/hitem3d/callback/route.ts`
  - 成功分支（`state='success'`）封面确定顺序：
    1. **供应商封面（cover_url）**
       - 若 `data.cover_url` 存在：
         - 尝试下载到：`assets/<user_uuid>/<asset_uuid>/cover.<ext>`；
         - `assets.cover_key = 该路径`；
         - 下载异常会被 catch 并打印日志，`cover_key` 清空以便后续 fallback。
    2. **用户上传输入封面（input cover）**
       - 当 `cover_key` 为空时：
         - 调用 `storage.listObjects({ prefix: 'assets/<user_uuid>/input-covers/<task_id>' })`；
         - 从结果中筛选第一张图片（`.png|.jpg|.jpeg|.webp`），并直接将该 key 作为 `cover_key`。
  - 创建资产时写入 `cover_key`：
    - `insertAsset({ uuid: asset_uuid, user_uuid: task.user_uuid, task_id, status: 'active', cover_key, file_key_full, ... })`。
  - 测试：`tests/integration/hitem3d-callback-route.test.ts`：
    - 原有用例验证供应商 cover → 下载 + `cover_key` 写入；
    - 新增用例验证“无 cover_url 时会从 `input-covers` 前缀中选一张图作为封面”。

- 列表展示封面：`/api/my-assets/overview` + `AssetCard`
  - 一旦 callback 成功创建资产，`cover_key` 就已就绪；
  - `getMyAssetsOverview` 为 `cover_key` 生成预签名 `cover_url`；
  - “我的资产”列表在首次展示时就有封面，不再依赖用户先打开 OBJ 详情页。

### 五、本次问题与修复说明（2025‑11‑15 晚）

> 现象：新生成资产在“我的资产”页面可以看到卡片封面，但 OBJ 预览长期停留在“OBJ 文件尚未就绪”，且 `/api/assets/:uuid/renditions/files?format=obj` 返回 `{"code":-1,"message":"files not available"}`（示例资产：`4a29d111-5b4f-4625-9475-798d209d3407`）。

- 根因分析
  - Step‑13 中为了“单卡片承载进度”，在 `app/api/hitem3d/submit/route.ts` 提交成功后，新增了“占位资产”写入 `assets` 表（仅有 `task_id + cover_key`），以便“我的资产”页面可以立即出现一张资产卡片。
  - `useTaskProgress` 的旧逻辑是：一旦 `/api/assets/by-task?task_id=...` 返回了 `asset_uuid`，就不再调用 `/api/hitem3d/finalize`。这在没有“占位资产”时是安全的，但引入占位资产后，会导致：
    - 任务状态 `state=success`；
    - 但由于已有资产记录，`finalize` 从未被真正触发；
    - `generation_tasks.hitem3d_file_url` 一直为空，`assets.file_key_full` 也不会被写入；
    - `/api/assets/:uuid/renditions/files?format=obj` 的 `materializeObjFiles()` 找不到 vendor `baseUrl`，只能返回 `files not available`，OBJ 永远不会 ready。

- 行为修复
  1. **资产卡片 + 任务进度绑定（前端）**
     - `components/assets/MyAssetsClient.tsx`：
       - 不再过滤 `has_asset=false` 的任务；改为仅做基于 `task_id` 的去重，将所有任务放入 `tasks`。
       - 通过 `taskById: Map<task_id, TaskOverview>` 将任务映射到资产，渲染时为每张资产卡片传入对应的 `task`：
         - `asset.task_id ? taskById.get(asset.task_id) : undefined`。
     - `components/assets/AssetCard.tsx`：
       - 新增可选 props：`task?: TaskOverview`、`disableProgressAutoPolling?: boolean`。
       - 内部使用 `useTaskProgress` 根据传入任务状态决定是否渲染进度条：
         - 当 `state ∈ {created, queueing, processing}` 时显示“预计进度 + 刷新状态”；
         - `state=success/failed` 时不再展示进度区块。
     - `components/assets/useTaskProgress.ts`：
       - 提取任务状态轮询 + 进度计算逻辑（原先在 `TaskCard` 中），供 `TaskCard` 与 `AssetCard` 复用。

  2. **占位资产与回调/兜底的一致性（后端）**
     - `app/api/hitem3d/submit/route.ts`：
       - `submitTask` 成功后，使用 `getUuid()` 生成 `asset_uuid`，写入占位资产：
         - `uuid/ user_uuid / task_id / status='active' / cover_key=assets/<user>/input-covers/<task_id>.<ext>`；
         - 这样“我的资产”中立刻有一张卡，并使用用户上传的输入图作为封面。
     - `app/api/hitem3d/callback/route.ts`：
       - 成功分支中，先 `findAssetByTaskId(task.task_id)`：
         - 若存在占位资产：复用其 `uuid`，调用 `updateAssetByUuid` 写入 `cover_key / file_key_full / file_format / updated_at`，避免重复创建资产；
         - 若不存在：按原逻辑 `insertAsset` 新建资产记录。
     - `app/api/hitem3d/finalize/route.ts`：
       - 同样在写入资产前先 `findAssetByTaskId`，优先填充已有占位资产的 `file_key_full`，避免重复资产。

  3. **总是执行 finalize 以确保供应商文件 URL 落库**
     - 旧逻辑：只在“任务 success 且仍未解析到资产 uuid”时才调用 `/api/hitem3d/finalize`，在有占位资产的场景下将永远不会触发。
     - 新逻辑（`components/assets/useTaskProgress.ts`）：
       - 在 `refreshOnce()` 中：
         - 无论 `/api/assets/by-task` 是否已返回 `asset_uuid`，只要 `effectiveState === 'success'`，都会执行一次：
           - `POST /api/hitem3d/finalize { task_id }`；
           - 成功后解析 `data.asset_uuid` 并记录本地 `assetUuid`。
       - 自动轮询逻辑取消了“`state=success && assetUuid` 即停止”的条件，只要在 1 小时窗口内且未失败，就会按每分钟刷新一次，确保 `finalize` 至少跑成功一次。
     - 影响：`generation_tasks.hitem3d_file_url` 将在供应商返回成功后稳定落库，为 OBJ/GLB 等派生流程提供稳定的 vendor 直链基础。

  4. **OBJ 文件未就绪时的 HTTP 语义调整**
     - `app/api/assets/[uuid]/renditions/files/route.ts`：
       - 当 OBJ 目录下仍然没有任何 `*.obj / *.mtl / 纹理`，且自动 materialize/解包 ZIP 均未成功时：
         - 之前：返回 `404 { code:-1, message:'files not available' }`，前端误认为“服务器不可用”；
         - 现在：返回 `202 { code:-1, message:'files not available' }`，语义为“文件正在准备中”（Accepted）。
       - 前端 `AssetAutoPreviewOBJ` 根据该状态，在 UI 中提示“OBJ 文件尚未就绪”，而不是“服务器异常”，并可通过“刷新状态”在稍后重试。
